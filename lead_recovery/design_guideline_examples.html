P1 — Consistent Icon Size

The icon core is always drawn at full canvas size. A badge-carrying icon and its solo counterpart have identical core dimensions. The badge is an additive overlay — it never causes the core to shrink, scale, or translate.

Rule: Solo icons and badged icons must have visually identical cores. The only structural difference between calls and answered-calls is the presence of a <g class="icon-badge"> element. The core path coordinates are byte-for-byte identical.

✗ WRONG

The scaled core also requires a compensated stroke-width='2.2' to maintain visual weight — an additional violation. Both problems disappear when the core is never scaled.

P2 — Badge as Additive Overlay

The badge sits on top of the icon in whichever corner has no icon core strokes. It does not push, move, or resize the core. If the preferred corner is occupied, the badge moves to the next free corner.

Rule: The badge is a small, independent overlay. The core never moves to accommodate it. Whichever corner of the 24×24 canvas has no icon strokes — that is where the badge goes.

P3 — Independent Layers & DOM Structure

Every icon — solo or badged — uses a strict two-group structure: <g class='icon-core'> and (if badged) <g class='icon-badge'>. This makes each layer independently addressable via CSS, JavaScript, and animation libraries.

Rule: All icon paths belong to either icon-core or icon-badge. No paths are allowed loose inside <symbol>. The groups are the public API of the icon.

<path d="M22 16.92…"/>
</g>

✗ WRONG

Corner Priority — TR → BR → TL → BL

Use the first corner whose 7×7 exclusion zone is completely empty — no strokes from any source (icon-core or icon-badge). Dashed red = zone occupied. Dashed green = zone empty = badge placed here.

Rule: Always try Top-Right first. A corner is only eligible if its exclusion zone contains zero existing strokes — from either icon-core or any already-placed icon-badge. A single line endpoint, a baseline, or a tick mark inside a zone blocks that corner just as firmly as the main body does. Never skip a free corner out of preference; never use an occupied corner.

P4 — Semantic Color

Color carries meaning, not aesthetics. Badge color is always independent of icon core color. The same icon core can carry different badges with different semantic colors.

Rule: A red badge always means failure/error. A green badge means success. Amber means warning. Grey means neutral/inactive. Badge color is set on <g class="icon-badge" stroke="<semantic-color>">, never derived from the core color.

Each badge uses its own semantic color. The teal core is constant. Badge color encodes the state.

✗ WRONG

Badge uses the same teal as the core. The × cross is invisible at small sizes and carries no semantic meaning through color.

P5 — Strokes Only · Uniform stroke-width

All icon paths use strokes, not fills. stroke-width is uniformly 1.5 for all icon paths and badge paths. No compensated weights.

Rule: stroke-width="1.5" everywhere. Set on the <g> group and inherited by children. The old compensated value 2.2 (from the scaled-core era) is obsolete and must never appear.
Stroke weight: uniform 1.5 vs compensated 2.2
✓ CORRECT
1.5 core / 1.5 badge
1.5 solo

stroke-width='1.5' on both groups. Core and badge weights match. Solo and badged cores are visually identical weight.

✗ WRONG
2.2 core / 1.5 badge

Core inside scale(0.68) uses stroke-width='2.2' as compensation. Stroke weight bloated at large sizes, mismatches badge.

P6 — No Invisible Mass

Badges carry no bounding shapes. No circles, rectangles, or filled containers behind the badge glyph. Every pixel of markup must be visible ink.

Rule: The badge group contains only the glyph strokes — no container shape behind it. Containers occlude the icon core at small sizes and add visual noise that fights the icon's rounded strokes.
Badge container: raw strokes vs circle container
✓ CORRECT

Raw polyline strokes only. No shape behind the badge. Clean at all sizes.

✗ WRONG

A circle with fill-opacity sits behind the checkmark. At small sizes it occludes the icon core and adds a halo that disrupts visual weight.

❌ Anti-pattern 1 — Scaling the Icon Core

Never use scale() or translate() on the icon core group to make room for a badge. The core must always be full canvas size.

Scaled core vs full-size core
✓ CORRECT

Core at full canvas size. Badge overlaid. No transform on core group.

✗ WRONG

transform='scale(0.68) translate(0,1.5)' on the core. Core is visibly smaller than a solo icon, breaking size consistency.

<!-- ❌ WRONG -->
<g class="icon-core" transform="scale(0.68) translate(0,1.5)" stroke-width="2.2">
<path d="M22 16.92v3 ... z"/>
</g>

<!-- ✅ CORRECT -->
<g class="icon-core" stroke="#4EC9B0" stroke-width="1.5">
<path d="M22 16.92v3 ... z"/>
</g>

❌ Anti-pattern 2 — Skipping a Free Higher-Priority Corner

Always use the highest-priority corner that is genuinely empty. Never use TR if it's occupied; never skip TR if it's free.

Skipping TR when it is empty
✓ CORRECT

Phone body leaves TR exclusion zone completely clear. Badge correctly placed at TR anchor (20,4).

✗ WRONG

Badge placed in TL even though TR zone has zero existing strokes. Wastes the default position and breaks visual consistency across the icon family.

<!-- ❌ WRONG: TL badge when TR zone is completely empty -->
<g class="icon-badge" stroke="#22C55E" stroke-width="1.5">
<polyline points="1.5,4 3.5,6.5 6.5,1.5" .../> <!-- TL anchor -->
</g>

<!-- ✅ CORRECT: TR badge — TR zone had no existing strokes -->
<g class="icon-badge" stroke="#22C55E" stroke-width="1.5">
<polyline points="17.5,4 19.5,6.5 22.5,1.5" .../> <!-- TR anchor -->
</g>

❌ Anti-pattern 2b — Badge in a Corner Already Occupied by Core Strokes

The exclusion zone check covers ALL existing strokes. A single endpoint, baseline tick, or secondary element inside a zone blocks it as firmly as the main body. Check every corner against every drawn path.

Rule: "Occupied" means any ink in the 7×7 exclusion zone — not just the visually dominant part of the icon. A bar's bottom endpoint, a waveform tick, or a baseline that grazes a corner all block that corner. Walk the priority order (TR → BR → TL → BL) and place the badge in the first zone with zero existing strokes.
Bar chart: right bar's bottom endpoint blocks BR — badge must go to TL

✗ WRONG
Bar bottom (18,20)
inside BR zone

Bar bottom endpoint (18,20) is inside the BR exclusion zone but badge is forced into BR anyway. Badge and core stroke visually collide.

The key insight: it is the bar's bottom endpoint — a secondary, non-dominant element — that blocks BR. The check must cover every coordinate of every path, not just the visually prominent part.

❌ Anti-pattern 3 — Loose Paths Without Named Groups

All paths must be inside either icon-core or icon-badge. Loose paths in the symbol cannot be targeted independently.

Group structure
✓ CORRECT

Two named groups. Each group is the public API: CSS and JS can select, animate, or override each independently.

✗ WRONG

Paths loose inside <symbol>. document.querySelector('.icon-badge') returns null. Animation and CSS overrides require knowing internal path indices.

<!-- ❌ WRONG: loose paths -->
<symbol id="answered-calls" ...>
<path d="..." stroke="#4EC9B0"/> <!-- which is core? which is badge? -->
<polyline points="..." stroke="#22C55E"/>
</symbol>

<!-- ✅ CORRECT: named groups -->
<symbol id="answered-calls" ...>
<g class="icon-core" stroke="#4EC9B0" stroke-width="1.5">
<path d="..."/>
</g>
<g class="icon-badge" stroke="#22C55E" stroke-width="1.5">
<polyline points="..." fill="none"/>
</g>
</symbol>

❌ Anti-pattern 4 — Badge Container (Bounding Shape)

Badges must never have a background shape. No circle, rect, or ellipse behind the badge glyph.

Badge container
✓ CORRECT

Raw strokes only. Badge is pure ink — no background. Reads cleanly at 16px and 64px.

✗ WRONG

Circle with fill-opacity=0.25 behind the checkmark. Creates a halo, occludes icon core, adds optical weight.

<!-- ❌ WRONG: container behind badge -->
<g class="icon-badge">
<circle cx="20" cy="4" r="4.5" fill="#22C55E" fill-opacity="0.25" stroke="#22C55E"/>
<polyline points="17.5,4 19.5,6.5 22.5,1.5" stroke="#22C55E" fill="none"/>
</g>

<!-- ✅ CORRECT: raw strokes, no container -->
<g class="icon-badge" stroke="#22C55E" stroke-width="1.5">
<polyline points="17.5,4 19.5,6.5 22.5,1.5" fill="none"
stroke-linecap="round" stroke-linejoin="round"/>
</g>

❌ Anti-pattern 5 — Compensated Stroke Weight

stroke-width='2.2' is a relic of the scaled-core era. The core is never scaled, so all paths use stroke-width='1.5' uniformly.

Stroke weight: 1.5 vs 2.2
✓ CORRECT

Uniform stroke-width='1.5' on both groups. Visual weight matches solo icon perfectly.

✗ WRONG

stroke-width='2.2' inside a scale(0.68) group. Bloated stroke at large sizes, incompatible with system.

<!-- ❌ WRONG: compensated stroke from scaled-core era -->
<g class="icon-core" transform="scale(0.68)" stroke-width="2.2"> ... </g>

<!-- ✅ CORRECT: uniform weight, no transform -->
<g class="icon-core" stroke-width="1.5"> ... </g>
<g class="icon-badge" stroke-width="1.5"> ... </g>

❌ Anti-pattern 6 — Status Dot Embedded in Icon Body

A status indicator (active, live, online) must be a proper badge in an icon-badge group. It must never be embedded inside the icon-core group at coordinates inside an exclusion zone.

Active call dot: badge vs embedded
✓ CORRECT

Green dot is <circle cx='20' cy='4' r='2'> inside a separate icon-badge group. CSS/JS can show/hide it independently.

✗ WRONG

Green dot at cx='19' cy='5' is inside the icon-core group, inside the TR exclusion zone. Cannot be independently toggled. Violates single-responsibility of the core group.

<!-- ❌ WRONG: dot inside core, inside exclusion zone -->
<g class="icon-core" stroke="#4EC9B0" stroke-width="1.5">
<path d="M22 16.92v3 ... z"/>
<circle cx="19" cy="5" r="2" fill="#22C55E"/> <!-- in exclusion zone, in core -->
</g>

<!-- ✅ CORRECT: dot as badge in its own group -->
<g class="icon-core" stroke="#4EC9B0" stroke-width="1.5">
<path d="M22 16.92v3 ... z"/>
</g>
<g class="icon-badge" fill="#22C55E">
<circle cx="20" cy="4" r="2" stroke="none"/> <!-- badge group, at anchor -->
</g>

❌ Anti-pattern 7 — Sub-half-unit Coordinates

All coordinates must snap to 0.5-unit grid. Values like 17.3, 4.1, 22.7 cause sub-pixel rendering artefacts, especially at 16–24px render sizes.

Coordinate precision
✓ CORRECT

All coordinates on the 0.5 grid: 17.5, 4, 19.5, 6.5, 22.5, 1.5. Crisp at any size.

✗ WRONG

Sub-pixel coords: 17.3, 4.1, 19.4, 6.6, 22.6, 1.4. Badge strokes may blur or feather at small render sizes.

<!-- ❌ WRONG: sub-half-unit coordinates -->
<polyline points="17.3,4.1 19.4,6.6 22.6,1.4" .../>

<!-- ✅ CORRECT: snapped to 0.5 grid -->
<polyline points="17.5,4 19.5,6.5 22.5,1.5" .../>

❌ Anti-pattern 8 — Same Color for Badge and Core

Badge color must be semantically distinct from the core color. A teal × cross on a teal phone is invisible and meaningless.

Badge color independence
✓ CORRECT

Teal core, red badge. The × is immediately visible. Red communicates 'missed/failed' unambiguously.

✗ WRONG

Teal badge on teal core. The × disappears into the background and carries no semantic meaning through color.

<!-- ❌ WRONG: badge inherits/reuses core color -->
<g class="icon-core" stroke="#4EC9B0" stroke-width="1.5"> ... </g>
<g class="icon-badge" stroke="#4EC9B0" stroke-width="1.5"> <!-- same teal — meaningless -->
<line x1="17.5" y1="1.5" x2="22.5" y2="6.5" stroke-linecap="round"/>
</g>

<!-- ✅ CORRECT: badge uses its own semantic color -->
<g class="icon-core" stroke="#4EC9B0" stroke-width="1.5"> ... </g>
<g class="icon-badge" stroke="#EF4444" stroke-width="1.5"> <!-- red = missed/failed -->
<line x1="17.5" y1="1.5" x2="22.5" y2="6.5" stroke-linecap="round"/>
</g>

❌ Anti-pattern 9 — Ignoring a Secondary Element That Occupies a Corner

The exclusion zone check is not a visual impression test. Check every anchor coordinate and every arc curve against all four zones. A single endpoint or arc bulge inside a zone blocks that corner regardless of how minor it looks.

Rule: For each corner, check every path coordinate in icon-core — including M start points, arc endpoints, and for A/a arcs also the curve itself. If any part of any path falls inside a corner's 7×7 exclusion zone, that corner is off-limits. Arc curves can bulge into a zone even when their start and end points are outside it.
Common secondary elements that silently occupy corners
Baseline endpoint in BR

Baseline endpoint (22,20) is inside BR zone (x>16.5, y>16.5). Circled. BR is blocked even though no bar touches it directly.

Arc curve bulges into TR

Arc endpoints at (22,9.94) and (18,10) are outside TR (y>7.5). But the arc curve bulges rightward and upward, visually passing through TR. Endpoint check alone is not enough for arcs — inspect the rendered curve.
